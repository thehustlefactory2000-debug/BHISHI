# Bhishi Admin PWA — Full PRD & Codex Prompt

## Project Overview

Build a **Progressive Web App (PWA)** called **Bhishi Admin** using **React** (Vite or CRA), **TypeScript**, **Tailwind CSS**, and **localStorage** (or Supabase if backend is needed). The app is admin-only — no member-facing UI. It manages rotating savings groups (called "Bhishi" in Marathi/Hindi culture), tracks monthly payments, calculates interest-adjusted amounts, selects winners, and locks completed months permanently.

---

## Tech Stack

- React 18 + TypeScript
- Tailwind CSS + shadcn/ui
- React Router v6 (client-side routing)
- Zustand or Context API for global state
- localStorage or Supabase for persistence
- PWA: `vite-plugin-pwa` with offline support, manifest, service worker
- i18n: `react-i18next` (English, Marathi, Hindi minimum)
- Theming: CSS variables, light/dark/system modes

## Data Models

```typescript
// Admin account
interface Admin {
  id: string;
  email: string;
  passwordHash: string;
  createdAt: string;
}

// Group
interface BhishiGroup {
  id: string;

  totalMembers: number;        // N
  interestRate: number;        // r (e.g. 0.01 = 1%)

}

// Payment per member per month
interface Payment {
  memberId: string;
  paid: boolean;
  paidDate?: string;           // ISO date
  paymentMode?: 'online' | 'offline';
  paidAmount?: number;
}
```

## Payment Formula

```
P(m) = A × (1 − r × (N − m))
```

- `A` = base monthly amount
- `r` = interest rate (entered by admin when creating group, e.g. 0.01)
- `N` = total months (= total members)
- `m` = current month number (1-indexed)
- **Month 1**: always full amount `A` (no deduction)
- **Month N** (last): always full amount `A` (no deduction)
- **Months 2 through N−1**: apply formula above

**Example** (A=500, N=12, r=0.01):
| Month | Formula | Payment |
|-------|---------|---------|
| 1 | full | ₹500 |
| 2 | 500×(1−0.01×10) | ₹450 |
| 3 | 500×(1−0.01×9) | ₹455 |
| 11 | 500×(1−0.01×1) | ₹495 |
| 12 | full | ₹500 |

Implement `calculatePayment(A, r, N, m)` as a pure utility function used everywhere.

### 6. Winner Selection Modal

Triggered from month card after all members paid.

**Step 1**: Shows list of eligible members (those who have `hasWon: false`) as selectable cards with name and phone.

**Step 2**: Admin taps a member → confirmation: "Confirm {name} as winner for Month {m}? They will receive the payout."

**Step 3**: On confirm:
- Set `member.hasWon = true`, `member.payoutMonth = m`
- Set `monthRecord.winnerId = memberId`
- Show payout summary: "Payout amount: ₹{totalCollected for this month}" (sum of all members' actual payments for that month)
- Lock month automatically or show "Lock Month" button

**Guards**:
- Block if any payment is unpaid → show error: "All members must pay before selecting a winner"
- Block if member already won → member not shown in list
- Block if month is already locked → modal not accessible

## Business Logic Rules (Enforce Strictly)

1. **Payment formula**: `calculatePayment(A, r, N, m)` — first and last month always return `A`, middle months use formula.
2. **Cannot select winner if any payment is unpaid** for that month.
3. **Each member can win at most once** across the entire group.
4. **Locked months are fully immutable** — no field changes, no payment edits, no winner changes.
5. **Auto-lock option**: after winner is selected, prompt admin to lock the month.
6. **Cannot edit group financials** (amount, member count, rate) after payments begin.
7. **Payout amount for winner** = sum of all actual payments made by members for that month.
8. **Group completes** when all N months are locked.

## Key Utility: `calculatePayment`

```typescript
export function calculatePayment(
  A: number,   // base amount
  r: number,   // interest rate as decimal (e.g. 0.01)
  N: number,   // total months
  m: number    // current month (1-indexed)
): number {
  if (m === 1 || m === N) return A;
  return Math.round(A * (1 - r * (N - m)));
}
```
